# Daily Double

A Jeopardy!-style trivia game where every clue is generated by AI. One fresh board per day —
two rounds (Jeopardy! and Double Jeopardy!, 60 clues, Daily Doubles included) — written by
Claude, and your typed answers are judged by Claude too (leniently, the way a human host would).

## How it works

- **Daily board, stored in Firestore.** `GET /api/board` returns the board for a date (today by
  default). The first request of the day generates it — a category call + parallel clue calls
  per round — and saves it to `jeopardyBoards/{date}` via the Firebase Admin SDK. Every visitor
  plays the same board, so generation cost is fixed per day regardless of traffic, and past
  boards stay playable forever.
- **Two rounds, doubled values, Daily Doubles.** Round 1 (Jeopardy!) is 6 categories × $200–$1000;
  Round 2 (Double Jeopardy!) is 6 harder categories × $400–$2000. Each round hides 1 (round 1) or
  2 (round 2, different categories) Daily Double clues — real-rules approximation in
  `src/lib/jeopardy.ts` → `placeDailyDoubles()`. Opening a Daily Double shows a wager prompt
  (min $5, max = your current score or the round's top value, whichever is higher) before the
  clue text is revealed; that clue scores the wager instead of its face value.
- **Answers never reach the browser.** The client gets clues and dollar values only (plus which
  clue is a Daily Double, so the client can hold the wager screen — see the comment on
  `PublicClue` in `src/lib/jeopardy.ts` for the trust tradeoff). Judging happens server-side:
  `POST /api/judge` looks up the answer in the stored board and asks Claude for a lenient ruling
  plus a one-line quip.
- **Leaderboards + percentile.** When a game ends, the player can post a name; `POST /api/scores`
  validates the payload and records name, score, right/wrong/passed counts, and completion time
  under `jeopardyBoards/{date}/scores`, keeping a denormalized `topScore` on the board doc for
  the archive list. The response includes a percentile computed via two Firestore aggregate-count
  queries, shown as a radial meter (`src/components/PercentileMeter.tsx`).
- **Sign in (Google or email) to track your scores.** `src/lib/firebaseClient.ts` +
  `AuthProvider`/`AccountBar`/`AuthModal` wrap the Firebase Auth *client* SDK — separate from the
  Admin SDK used everywhere else. A signed-in submission includes a Firebase ID token; the server
  verifies it (`firebase-admin/auth`) and mirrors the score to `users/{uid}/scores/{date}` (one
  doc per user per day) so `/me` can list your history with a plain collection read — deliberately
  not a Firestore `collectionGroup` query, which would need a manually-created composite index.
- **Archive.** `/boards` lists every stored board (categories + top scorer) with a link to that
  date's full leaderboard (`/boards/[date]/scores`); `/boards/[date]` replays one.
- **Progress persists locally.** Score, round, and answered clues are stored in `localStorage`,
  keyed to the board's date, so a reload resumes the same game.
- Model: `claude-opus-4-8` with structured outputs (JSON schema) for boards and rulings.

## Local development

```sh
cp .env.example .env.local   # add ANTHROPIC_API_KEY, Firebase Admin credentials, and the
                              # public NEXT_PUBLIC_FIREBASE_* client config (see .env.example)
npm install
npm run dev
```

Open http://localhost:3000. The first load generates the day's board (~30–60s across two
rounds); after that it's served from Firestore.

**Firebase Console one-time setup for auth:** Authentication → Sign-in method → enable **Google**
and **Email/Password**. Authentication → Settings → Authorized domains → add your deployed
domain (`localhost` is pre-authorized for local dev).

## Deploying (AWS Amplify Hosting)

1. Push this repo to GitHub and connect it in Amplify Hosting (framework: Next.js — SSR,
   platform `WEB_COMPUTE`).
2. Add environment variables in Amplify → App settings → Environment variables:
   `ANTHROPIC_API_KEY`, `FIREBASE_SERVICE_ACCOUNT_BASE64` (run
   `base64 -i serviceAccountKey.json | tr -d '\n'` and paste the output), and the six
   `NEXT_PUBLIC_FIREBASE_*` client-config values (these are public by design — see
   `.env.example`). The included `amplify.yml` writes them to `.env.production` during the build
   so the SSR runtime can read them; `NEXT_PUBLIC_*` values also need to be present at *build*
   time since Next.js inlines them into the client bundle.
3. Deploy. **No domain purchase needed** — add a subdomain of an existing Route 53-hosted domain
   (e.g. `dailydouble.stanleyluong.com`) via Amplify Hosting → Domain management → Add domain;
   Amplify provisions the cert and gives you the CNAME/ALIAS record to add in Route 53.
4. Add that domain to Firebase Console → Authentication → Settings → Authorized domains, or
   Google/email sign-in will fail on the live site even though it works locally.

## Notes

- The in-memory rate limiter (`src/lib/rateLimit.ts`) is per-instance and best-effort — fine for
  a low-stakes game whose expensive call happens once a day, but not a hard guarantee.
- **Firestore security rules:** all reads/writes go through the Admin SDK (which bypasses
  rules), so make sure your rules do **not** grant clients access to `jeopardyBoards` — the
  stored boards include the answers. `users/{uid}/scores` should be readable/writable only by
  that uid if you ever add client-side Firestore access there (currently everything goes through
  the Admin SDK, so this is a defense-in-depth note, not a current gap).
- Score submissions are validated for shape and plausibility (score range/granularity, counts,
  duration) but the game state itself lives client-side, so the leaderboard is honor-system —
  appropriate for a portfolio game, not for prizes. Wager amounts are similarly client-trusted
  (not re-validated server-side against the player's actual running score).
- The judge endpoint answers `409 board-changed` if a client's board no longer matches the
  stored one (e.g. a race on the day's first generation); the client reloads automatically.
